## 1. Fixes

### 1.1 Watchtower autoupdate 호환성

autoupdate profile (`docker compose --profile autoupdate up -d`)이 최신 Docker 호스트에서 동작하지 않았다. 게시되어 있던 watchtower 이미지가 archived 상태였고, 컨테이너 내장 Docker client가 호스트 daemon과 통신할 수 없었던 것이 원인.

**증상:**
- `docker logs nas-watchtower`가 polling 주기마다 `client version 1.25 is too old. Minimum supported API version is 1.44`를 출력.
- 부수효과: 호스트의 모든 `docker` 명령도 `WARNING: Error parsing config file (...): is a directory` 경고를 함께 출력.
- 결과: autoupdate가 실제로 동작하지 않음 — 이미지 pull이 시도조차 되지 않음.

**원인:**
- `containrrr/watchtower:latest`는 2023-11-11 이후 재빌드되지 않았고 upstream 리포지토리가 archived. 내장 Docker client API 버전(v1.25)이 Docker 24.0 이상 daemon이 요구하는 v1.44 floor 미달.
- compose 파일이 `${HOME}/.docker/config.json`을 컨테이너 안으로 bind-mount. 해당 파일이 호스트에 없을 경우(README의 public-GHCR 안내를 따른 사용자에게 흔한 상황) daemon이 mount source 위치에 root-소유 빈 디렉토리를 자동 생성. watchtower는 이걸 config 파일로 파싱하려다 실패하고, 같은 경로가 디렉토리로 인식되는 부작용이 이후 호스트 `docker` 호출마다 경고를 발생.

**수정:** (commit `8641c76`, PR #1로 merge → `0dfa2eb`)
- `docker-compose.yml`: watchtower 이미지를 `nickfedor/watchtower:latest`로 전환. 활성 유지보수 중인 커뮤니티 fork이며 nightly 빌드와 최신 Docker client(Docker 29.x daemon 상대로 Docker API v1.51 negotiate)를 제공.
- `docker-compose.yml`: `${HOME}/.docker/config.json:/config.json:ro` bind mount 제거. README 기본값인 public GHCR 환경에서는 watchtower 컨테이너의 레지스트리 인증이 필요하지 않음. private 레지스트리를 사용하는 운영자는 호스트에서 `docker login ghcr.io`로 처리.

수정 후 `docker logs nas-watchtower`는 `Watchtower ... using Docker API v1.51`과 정상 `Next scheduled run` 주기를 출력하며, 호스트의 docker 경고도 사라진다.

## 2. Infrastructure and Refactors

- **stable 단계에 맞춘 CI 메타데이터 정책 전환** (commit `3938702`). `.github/workflows/build-and-deploy.yml`에서 `0.0.1-beta` 단계 동안 임시로 적용했던 `latest=true`를 `flavor: latest=auto`로 되돌리고, `type=semver,pattern={{major}}.{{minor}}` 태그 패턴을 다시 활성화. `v0.1.0`부터의 태그 push 동작:
  - 가장 높은 non-prerelease 태그가 `:latest`로 publish됨. 향후 pre-release (`v0.2.0-rc.1`, `v0.1.1-beta` 등)는 `:latest`로 승격되지 않아 `:latest`가 stable 이동 태그로 유지됨.
  - 각 stable 태그는 minor-pin alias도 함께 publish (예: `ghcr.io/jmc50/nas:0.1`). consumer가 minor line에 pin해두면 patch 릴리스를 자동으로 따라가되 minor boundary는 넘지 않음.
- 워크플로우 트리거 모델과 Dockerfile 경로는 변경 없음 (`v*` 태그 push + `workflow_dispatch`, `./backend/Dockerfile`로 빌드).

## 3. Verification Summary

Watchtower 이미지 교체는 운영 배포 호스트(Ubuntu 26.04 LTS / Docker 29.1.3 / Server API v1.52)에서 검증. `nickfedor/watchtower` 컨테이너가 Docker API v1.51로 negotiate하고, 오류 없이 polling하며, daemon이 거부하던 `containrrr/watchtower` v1.25 client를 대체. bind-mount 제거로 호스트 `docker` 호출의 경고도 사라짐.

본 릴리스에서 frontend/backend 소스 경로는 변경되지 않아 기존 정적 검사·빌드·통합 테스트는 `v0.0.1-beta`와 동일.

CI 메타데이터 정책 전환은 구조적 변경 — `flavor: latest=auto`와 `type=semver,pattern={{major}}.{{minor}}`는 `docker/metadata-action`의 문서화된 설정 — 이며, 결과는 `v0.1.0` 태그 push 이후 GHCR 태그 목록에서 관찰됨:

| 태그 | 출처 |
|---|---|
| `:0.1.0` | `type=semver,pattern={{version}}` |
| `:0.1` | `type=semver,pattern={{major}}.{{minor}}` (이번 릴리스에 신규 활성) |
| `:latest` | `flavor: latest=auto` (highest non-prerelease가 자동 promote) |

모든 태그는 기존 `docker/build-push-action` multi-arch 설정에 따라 `linux/amd64`와 `linux/arm64` 양쪽으로 publish된다.

## 4. Known Limitations and Follow-ups

- `v0.0.1-beta` 태그는 pre-release 이력으로 그대로 보존. 대응하는 GHCR `ghcr.io/jmc50/nas:0.0.1-beta` 이미지도 해당 버전 pin이 필요한 운영자를 위해 그대로 유지. 신규 배포는 `:latest`, `:0.1`, 또는 `:0.1.0`을 추적하면 됨.
- 본 릴리스는 애플리케이션 기능 변경 없음. 0.1.x 라인은 stability/policy marker로 시작하며, 예정된 기능 작업(추가 viewer 타입, mobile 인터랙션 정련, 검색 동선 개선)은 별도로 트래킹되어 향후 minor/patch 릴리스에 반영 예정.
- Watchtower upstream 유지보수는 커뮤니티(`nickfedor` fork) 주도. supply-chain 요구가 엄격한 운영 환경은 `:latest` 대신 digest pin 사용 권장.
