v0.1.0 is the first stable cut of the project, following the `v0.0.1-beta` pre-release. The delta is intentionally narrow — two bugfixes (autoupdate compatibility and frontend version display) and one CI/CD metadata policy switch to align with stable semver conventions. No frontend or backend source paths changed since `v0.0.1-beta`.

---

## 1. Fixes

### 1.1 Watchtower autoupdate compatibility

The autoupdate profile (`docker compose --profile autoupdate up -d`) was broken on modern Docker hosts because the published watchtower image was archived and its embedded Docker client could no longer talk to the host daemon.

**Symptoms:**
- `docker logs nas-watchtower` printing `client version 1.25 is too old. Minimum supported API version is 1.44` on every poll cycle.
- Side effect: every host `docker` invocation also printing `WARNING: Error parsing config file (...): is a directory`.
- Net result: autoupdate did not function — image pulls were silently never attempted.

**Root cause:**
- `containrrr/watchtower:latest` has not been rebuilt since 2023-11-11 (the upstream repository is archived). Its embedded Docker client is below the v1.44 floor required by Docker 24.0 daemons and newer.
- The compose file bind-mounted `${HOME}/.docker/config.json` into the container. When that file does not exist on the host (the common case for public-GHCR users following the README), the daemon auto-creates a root-owned empty directory at the mount source. Watchtower then fails to parse it as a config file, and the same path being treated as a directory triggers the warning in every subsequent host `docker` call.

**Fix:** (commit `8641c76`, merged via PR #1 as `0dfa2eb`)
- `docker-compose.yml`: watchtower image switched to `nickfedor/watchtower:latest` — the actively maintained community fork, with nightly builds and a current Docker client (negotiates Docker API v1.51 against Docker 29.x daemons).
- `docker-compose.yml`: removed the `${HOME}/.docker/config.json:/config.json:ro` bind mount. Public GHCR (the default per README) requires no registry auth in the watchtower container. Operators using a private registry should run `docker login ghcr.io` on the host instead.

After this fix, `docker logs nas-watchtower` shows `Watchtower ... using Docker API v1.51` followed by a normal `Next scheduled run` cadence; the host-side docker warning disappears.

### 1.2 Frontend version display showed `0.0.0-unknown`

The StatusBar rendered `v0.0.0-unknown` instead of the actual release version in every published container image since `v0.0.1-beta`.

**Root cause:** `frontend/vite.config.ts` injects `__APP_VERSION__` at build time by reading the repo-root `VERSION` file (`resolve(__dirname, "..", "VERSION")` → `/work/VERSION` inside the build container). The `backend/Dockerfile` frontend-build stage copied `frontend/` but never copied `VERSION`, so `readFileSync` threw and the `catch` fallback `"0.0.0-unknown"` became the inlined constant. Local `npm run dev` and `npm run build` were unaffected because the host filesystem has the `VERSION` file at the expected location — the bug was containerization-specific.

**Fix:** `backend/Dockerfile` adds `COPY VERSION ./` to the frontend-build stage before `npx vite build`. The injected constant now reflects the canonical `VERSION` value, so this release renders `v0.1.0` in the StatusBar (and any future release will render its tag value).

## 2. Infrastructure and Refactors

- **CI metadata policy reverted for the stable phase** (commit `3938702`). `.github/workflows/build-and-deploy.yml` switches `flavor: latest=auto` back from the temporary `latest=true` used during the `0.0.1-beta` phase, and re-enables the `type=semver,pattern={{major}}.{{minor}}` tag. Behavior on tag push from `v0.1.0` onward:
  - Highest non-prerelease tag is published as `:latest`. Future pre-releases (`v0.2.0-rc.1`, `v0.1.1-beta`, etc.) will not be promoted to `:latest`, preserving it as the stable moving tag.
  - Each stable tag also publishes a minor-pin alias (e.g. `ghcr.io/jmc50/nas:0.1`) so consumers can pin a minor line and auto-follow patch releases without crossing minor boundaries.
- Workflow trigger model and Dockerfile path are unchanged (tag push on `v*` + `workflow_dispatch`, build from `./backend/Dockerfile`).

## 3. Verification Summary

The Watchtower image swap was validated against the production deployment host (Ubuntu 26.04 LTS / Docker 29.1.3 / Server API v1.52). The `nickfedor/watchtower` container negotiates Docker API v1.51, polls without error, and replaces the daemon-rejected `containrrr/watchtower` v1.25 client. The bind-mount removal eliminates the warning from every subsequent host `docker` invocation.

No frontend or backend source paths changed in this release, so existing static checks, builds, and integration suites are unchanged from `v0.0.1-beta`.

The CI metadata policy switch is structural — `flavor: latest=auto` and `type=semver,pattern={{major}}.{{minor}}` are documented `docker/metadata-action` configurations — and its effect is observable in the GHCR tag list after the `v0.1.0` tag push:

| Tag | Source |
|---|---|
| `:0.1.0` | `type=semver,pattern={{version}}` |
| `:0.1` | `type=semver,pattern={{major}}.{{minor}}` (newly enabled) |
| `:latest` | `flavor: latest=auto` (now driven by highest non-prerelease) |

All tags publish on both `linux/amd64` and `linux/arm64` via the existing `docker/build-push-action` multi-arch configuration.

## 4. Known Limitations and Follow-ups

- The `v0.0.1-beta` tag is retained as a historical pre-release marker. The corresponding GHCR `ghcr.io/jmc50/nas:0.0.1-beta` image stays available for operators who need to pin to it; new deployments should track `:latest`, `:0.1`, or `:0.1.0`.
- This release ships no application feature changes. The 0.1.x line begins as a stability/policy marker; planned feature work (additional viewer types, mobile interaction refinements, search ergonomics) is tracked separately and will land in future minor/patch releases.
- Watchtower upstream maintenance is community-driven (`nickfedor` fork). Operators with strict supply-chain requirements may prefer to pin to a digest rather than `:latest`.
